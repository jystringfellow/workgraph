LLM Integration Specification

Internal Document

Context

To empower workgraph as a tool for personal work intelligence, it must be able to leverage Large Language Models (LLMs) for tasks such as summarization, categorization, and proactive insight generation.

While the system is "Local-First," we recognize that some users prefer high-capability hosted models (OpenAI/Anthropic), while other users require more private "Cloud" options (AWS Bedrock/Azure Foundry).

Principles

  • Provider Agnosticism: The core business logic of workgraph must not know which provider it is talking to. It should interact with a unified LLMProvider interface.
  • Data Sovereignty: Only the data required for the specific task (the "context") should be sent to any external API.
  • Local-First Continuity: The absence of an LLM or a failed connection to one must not break core functionality. LLM features are "enhancements."

Architecture: Provider Pattern

We will implement a provider-based abstraction to handle the differences between local inference, cloud profiles (Bedrock/Azure), and standard API keys.

1. The Unified Interface

All integrations must satisfy a common interface in internal/llm:

  • Generate(ctx, request) -> Response
  • Stream(ctx, request) -> Channel (Optional for real-time UI)

2. Supported Categories

The system should support three categories of providers:

  • Standard API: Standard HTTP calls to OpenAI, Anthropic, etc., requiring an API Key.
  • Cloud Inference Profiles: Integration with AWS Bedrock and Azure Foundry, which allow the user to use high-powered models within their own account/tenant.
  • Local Bridge: Support for local inference engines (e.g., Ollama) via a standard OpenAI-compatible local endpoint.

Data Flow & Context Management

To preserve privacy and manage token costs, we will implement the following:

  • Just-In-Time (JIT) Context Gathering: The system should not send an entire database dump. A Context Fetcher identifies relevant records in SQLite based on dates or tags to construct a focused prompt.
  • Prompt Templates: Prompts are defined as static templates with placeholders (e.g., "Summarize the following notes from {{date}}: {{content}}") stored in a dedicated directory.
  • Handling Latency: All LLM calls must be performed asynchronously or via worker threads to ensure the UI/CLI remains responsive.

Error Handling & Fallbacks

  • Partial Success: If an LLM is unavailable, the system should still save the raw data and provide a "Retry" option for the AI operation later.
  • Validation: The output of any LLM call must be validated before being persisted as an "official" record in the system.
